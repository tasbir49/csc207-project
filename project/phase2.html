Thoughts about design and about the project

Read this handout

We spent hours and hours writing this handout so we clearly think it's important. Here is why you should read it carefully:

  • You are all going to work for hours and hours on it and we want you to understand the requirements before you start writing code.
  • If you want to program for a living, you need to work on your software design and development skills. This project is modelled after a real-world project that actually happened.

You and your group need to work hard, and you need to work together. If you can't agree on something, drop by office hours or post on the discussion board to get advice. Get it done one way or another: programs that run always get more marks than those that don't. Don't bully each other, though: talking to us about it might reveal an idea that is even better than your current options.

System boundaries

Something to think about: every system has communcation boundaries where it interacts with other devices. As you've discovered in phase 1, your system has boundaries where it interacts with the real world. In the warehouse, one boundary is the communication with the barcode readers. Another is the place — probably a computer — where orders are entered. A third is the warehouse traversal optimization. Are there any others?

When designing a system, identifying the boundaries is of course quite important, not only for design considerations but also because they should appear in the contract between the warehouse and the software developers.

Design for testing

Those of you who actively thought about software testing as you did your design did yourselves a service. Trying to write tests for a system after it has been implemented can be quite frustrating (although it's worthwhile).

Testing is a requirement for phase 2. If you haven't started writing and running unit tests as you develop, you should start phase 2 by writing a basic set of unit tests. Design for extensibility

In the real world, if someone pays you do this project, then you have a product that other warehouses might want. Hopefully you own or have rights to the Intellectual Property (IP), so that you might sell it to other warehouses. Naturally, other warehouses will be configured differently, so your software should be useable elsewhere with as little modification as possible, in the extreme just by providing new configuration files. We'll be marking you on this.

Testing is a requirement for phase 2. If you haven't started testing, you should start phase 2 by writing as many tests as you can.

What to do for Phase 2

Refactoring

During Phase 2, refactor proactively. Nobody owns the code in your group: everyone should feel free (after confirming with their group that refactoring is a good idea) to go in and edit any code. As you refactor, edit the test cases. Incorporate any design patterns that you think are appropriate.

Here are some major points that the TAs will be looking for. These will also help get your test coverage up as high as possible.

  • All your variables should be private, and you should only provide setters and getters if other classes actually need the information.
  • Ideally, the only static member should be method main. If you have a static variable, you need to write a comment explaining why: is it a shortcut to having working that you recognize isn't perfect? Is it because you believe that even in a general setting you won't need more than one?

Testing

Install EclEmma from the Eclipse Marketplace. The higher your test coverage, the higher your project mark. You should aim for as close to 100% as you can, because your raw mark will be multiplied by your code coverage.

We strongly recommend that you start Phase 2 by setting up some basic unit tests and making sure that everyone in the group can run them.

Files from Phase 1

Update the duties.txt, help.txt, and design.pdf files to reflect only Phase 2 work. What have you contributed to the team since you first read this handout? Remember, every group member needs to write their own.

Logging

In Phase 1, many groups used print statements to simulate your system sending messages to workers through barcode readers. For Phase 2, use both a ConsoleHandler and a FileHandler (as discussed in lecture). You need to log at least these things to both the console and to project/log.txt:

  • The events
  • The messages that the system sends to the barcode readers
  • The messages the barcode readers sends to the system
  • The lines from the event input file

    • In the log, it should be clear what the input events to you system are, and what the output events are.

Requirements for the event file

For Phase 2, your program must have the following features (which may differ from how you did Phase 1, but are now required):

  • Your program should take the name of an event file (like 16orders.txt) as a command-line argument, so that filname will be at index 0 in main's parameter args. On the command line, the command will look something like this: java warehouse/Warehouse ../16orders.txt. Notice that the argument is the relative path to the event file. In Eclipse, to add such an argument, right-click on the project, select Run As -> Run Configurations, click the Arguments tab, and type ../16orders.txt (or whatever file you want) in the Program Arguments box. Everyone should learn how to do this, so we suggest figuring it out as your group. We are considering whether to ask questions about it on the final exam.
  • Every worker (pickers, sequencers, loaders, and replenishers) should have an event announcing the worker is ready. The 16orders.txt file currently only has this kind of event for pickers, so you'll have to add the others if you haven't already.
  • There should be a "rescan" event at all times for having a sequencer or a loader re-try the the barcode scanning. Something like "Sequencer Sue rescans" for when Sue wants to start over scanning the SKUs, and then the simulation has events for Sue to start again with the first SKU on the sequenced pallets.
  • A picking event (the ones that look like "Picker Paul pick 1") represents the event where a picker uses their barcode scanner on a fascia barcode, after putting it on the forklift. For Phase 2, the number is the SKU of the fascia. Your simulation should read that event from the event file and tell your system that it happened. Your system should respond by checking the SKU against what it told the picker to pick. Notice that this means that the system tells the picker to get one SKU at a time, rather than sending all 8 at once.
  • Sequencers must use their barcode reader on all 8 fascia, one at a time, and the system must respond with a message saying whether the number is correct. Notice that this means that there are 8 sequencing events for each set of 4 orders. When all 8 have been confirmed, the system must tell a loader that it's time to load.
  • Loaders must have similar events to scanners because they rescan everything, and the system should confirm them.
  • SKUs may be many digits long, and they may include letters.
  • Pickers may make mistakes. If they scan the wrong SKU, your system should tell them that, and in the same message repeat the instruction with the correct SKU. Note that this means that a picker may be told to pick something more than 8 times in order to get 8 fascia.
  • Even with that rule, pickers may arrive at the marshalling area with an incorrect fascia. If the sequencer detects this, remember that the whole set get thrown out, and the system should tell the picker to get a replacement set of 8 fascia. Notice that this means that no loaders can load until this is fixed.

Presentation

Software Walkthrough: "A static analysis technique in which a designer or programmer leads members of the development team and other interested parties through a software product, and the participants ask questions and make comments about possible errors, violation of development standards, and other problems." (IEEE Std. 1028-1997, IEEE Standard for Software Reviews, clause 3.8)

The Phase 2 marking will take place in the last two weeks of class. (We will post a signup sheet on the web very soon.) You and your group will present your project to 2 TAs as if it were a software walkthrough. They may ask questions while you present, or they may wait until after the presentation. Your presentation should last no more than 15 minutes, plus at least 5 minutes for TA questions. Each of you may be asked about any of the code, including the test cases, design patterns, UML, and other implementation details.

Every group members must present for at least 3 of the 15 minutes. Plan carefully.

By the Phase 2 due date, you must commit a file called project/walkthrough.pdf that contains an outline of your presentation. You should cover at least these points:

  • What are the most important classes in your program?
  • How is the event file processed and what happens after you read an event? Everyone should be able to explain the series of methods that get called. We're going to ask you to do this in the interview, and your individual grade will depend on it. This is worth 25% of the project mark.
  • What data are you keeping track of, and how are you keeping track of it?
  • What design patterns did you use, if any? What problems do each of them solve?
  • What is your unit test coverage?

When you present, you will be expected to discuss your code in detail with the TAs as you walk through your system.
